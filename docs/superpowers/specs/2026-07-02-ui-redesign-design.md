# 《吃点啥》原生 Android UI 重设计方案

## 背景

当前 App 已具备做饭推荐、附近餐厅、收藏、设置、开发者模式、真实定位和地图跳转等功能，但界面仍偏早期原型：暖色卡片堆叠较多，视觉层级保守，动效弱，不能充分体现 Jetpack Compose 原生实现相对 HTML 原型的优势。

本次重设计方向已确认：使用方案 A 的明亮食欲配色，结合方案 C 的轻未来原生动效。目标是让 App 更年轻、更有食欲，同时在 AI 生成、定位、排序、收藏、页面切换等关键动作中呈现原生 Android 的流畅手感。

## 目标

- 建立新的视觉系统：明亮、潮流、有食欲，保留温暖但去掉老式米色卡片感。
- 加入轻未来 AI 氛围：动态高光、流光进度、定位雷达、结果错峰入场。
- 保持业务稳定：不改后端接口、不改 AI/高德/万维直连逻辑、不改数据模型含义。
- 提升原生感：使用 Compose 动画、触控反馈、状态转场和系统能力，而不是静态 HTML 风格页面。
- 改善代码结构：从单个 `MainActivity.kt` 中拆出主题、通用组件、页面组件和动效组件，降低后续维护成本。

## 非目标

- 不重做后端、菜谱数据库、AI 提示词或高德/万维 API 接入。
- 不新增账号体系、云同步、支付、地图 SDK 或复杂离线缓存。
- 不做深色赛博风全局主题；黑底霓虹只作为局部加载或 AI 状态点缀。
- 不追求 Material 3 默认模板感，仍以“吃什么”这个产品的美食氛围为主。

## 视觉系统

### 色彩

采用 A 方案的食欲配色作为全局基底：

- 背景：暖白、轻奶油白，替代现在偏旧的深米色。
- 主色：番茄红，用于主按钮、搜索提交、收藏成功和关键状态。
- 高光：阳光橙、暖黄，用于 Hero、AI 生成光带、食物氛围。
- 辅色：薄荷绿，用于附近、定位、健康、地图数据状态。
- 科技点缀：少量蓝紫或青色，只出现在 AI 状态、流光边框、雷达波纹中。

页面应避免单一色相铺满。番茄红不做大面积纯色背景，更多作为动作色；Hero 和按钮可以使用橙红黄渐变，但普通卡片保持清爽。

### 形状与层级

- 主要卡片圆角控制在 18-24dp，按钮和小标签圆角更圆，但不把所有内容都做成厚重卡片。
- 页面使用“无框分区 + 关键卡片”的结构，减少卡片套卡片。
- 首页、详情页使用大图或渐变 Hero 承担第一视觉，不再只有标题和小卡片。
- 列表卡片使用图片、标题、标签、评分/距离等信息的紧凑结构，适合反复扫读。

### 字体与信息密度

- 首页品牌和 Hero 标题可以更有冲击力。
- 列表、详情、设置保持中等字号，避免大标题挤压可用内容。
- 中文文案保留生活感，例如“今天适合吃点热的”“附近可出发”“AI 正在推演这一桌饭”。

## 动效系统

动效采用 C 方案：轻未来、克制、服务状态表达。

### 关键动效

- 页面切换：使用 `AnimatedContent` 做轻微滑入与淡入，底部导航切换不生硬闪屏。
- 首页 Hero：进入时轻微上浮，背景高光缓慢漂移，表现“今天吃什么”的活力。
- 搜索提交：按钮按下有缩放回弹，搜索面板可 `animateContentSize` 展开状态说明。
- AI 生成：显示流光进度条、食材标签流动、可取消按钮；长等待时保留已用时。
- 数据库搜索：显示较轻的脉冲扫描，不假装 AI 生成。
- 定位：点击定位后显示雷达脉冲，成功后位置卡片高亮一次。
- 结果列表：卡片错峰淡入/上移，排序切换时列表内容平滑重排。
- 收藏：心形图标弹性放大回落，可配合轻触觉反馈。

### 原生优势表达

- 使用 Compose 状态驱动动画，而不是网页式 CSS 假动效。
- 使用 `LocalHapticFeedback` 给收藏、搜索提交、定位成功提供轻触感。
- 使用 Android 权限、定位、地图 Intent 等系统能力，并在 UI 中给出即时状态动画。
- 使用可取消的协程加载状态，让 AI 生成等待过程可控。

### 动效约束

- 所有动画要可读、短促，不影响主流程。
- 无限动画只用于加载、雷达、背景高光等必要状态，避免耗电和分散注意。
- 卡片入场错峰控制在 200-450ms 内，避免列表长时间等待。

## 页面设计

### 首页

首页改为“今天吃什么控制台”：

- 顶部显示产品名、当天语气文案和设置入口。
- 第一屏放沉浸式食欲 Hero，展示当前推荐方向、AI/数据库/高德能力状态。
- Hero 下方是一个主搜索面板，可快速进入自己做或附近吃。
- 两个主入口使用更强的视觉区分：自己做偏番茄橙，附近吃偏薄荷绿。
- 今日灵感改为横向/纵向混合的精致卡片，不再像普通列表。

### 在家做推荐页

- 顶部保留返回和刷新，但标题区更轻。
- 搜索卡片突出“数据库 / AI 生成”双模式，按钮语义随模式变化。
- AI 摘要卡片改成轻未来状态条，显示解析结果、来源和标签。
- 组合菜单卡片展示整桌结构，菜品行更像“菜单排布”而不是普通文字列表。
- 单道菜卡片展示星级、来源、耗时和食材亮点。
- AI 生成中使用流光与可取消状态；数据库搜索中使用短脉冲扫描。

### 附近吃页面

- 顶部定位按钮触发雷达动效。
- 位置卡片显示真实地址/地标、坐标来源和状态提示。
- 搜索与筛选合并为“附近雷达”区域，排序方式作为可见 segmented 控件。
- 餐厅卡片突出真实 POI 来源：名称、距离、评分、价格、地址摘要、标签。
- fallbackReason 显示为“地图数据状态”，配合薄荷/橙色状态样式，不像错误弹窗。

### 详情页

- 菜谱、组合菜单、餐厅详情统一使用大图 Hero + 半透明信息条。
- 信息块改为更清爽的分区，不全部使用厚边框卡片。
- 餐厅详情的“导航去这里”使用醒目的底部主按钮，强调系统地图跳转。
- 菜谱步骤使用编号时间线，食材用紧凑标签格。

### 收藏与设置

- 收藏页改为更轻的聚合列表，支持分类筛选动效。
- 设置页保留偏好编辑，但减少卡片密度。
- 开发者模式继续保持工具感，禁用状态明确置灰；后端地址仍始终可编辑。

## 组件与代码结构

建议在 `app/src/main/java/com/dinnerplan/chidian/` 下拆分：

- `ui/theme/ChiDianTheme.kt`：色彩、Typography、Shape、全局 MaterialTheme。
- `ui/components/FoodScaffold.kt`：基础 Scaffold、底部导航、页面容器。
- `ui/components/FoodCards.kt`：Hero、推荐卡片、餐厅卡片、信息块。
- `ui/components/FoodControls.kt`：搜索框、模式切换、标签、排序控件、按钮。
- `ui/components/FoodMotion.kt`：流光、雷达、脉冲、错峰入场、点击缩放。
- `ui/screens/HomeScreen.kt`
- `ui/screens/CookRecommendScreen.kt`
- `ui/screens/NearbyRestaurantScreen.kt`
- `ui/screens/DetailScreens.kt`
- `ui/screens/SavedSettingsScreens.kt`

第一阶段不强制迁移所有数据模型，只把 Composable 和主题从 `MainActivity.kt` 拆出。业务状态和仓库接口可以暂时保留在现有文件中，后续再拆 ViewModel。

## 数据流与状态

- 现有 `AppUiState` 继续作为 UI 状态来源。
- `developerSettings.enabled` 继续决定后端代理或直连模式。
- `cookSourceMode` 继续决定数据库搜索或 AI 生成。
- `restaurantSortMode` 继续决定餐厅列表排序。
- 新增 UI-only 状态应局限在 Composable 内，例如卡片入场、定位动画、收藏反馈，不写入 DataStore。
- 不新增后端字段；现有 fallbackReason、loading、locationSource、ratingStars 等字段继续复用。

## 错误与空状态

- AI/万维/高德失败继续展示真实原因，但用更清爽的状态卡片表达。
- 空结果状态要提供下一步动作，例如重新搜索、换关键词、切换来源、检查开发者设置。
- 定位不可用时保留手动位置入口，不阻断附近搜索。
- 开发者模式缺少 Key 时，状态卡片明确指出是 AI、高德还是万维配置缺失。

## 可访问性与性能

- 所有图标按钮保留 contentDescription。
- 文字不使用 viewport 缩放；确保小屏中文不溢出。
- 动效时间短，加载动画可取消或随状态停止。
- 图片继续使用 Coil，避免一次性加载过大资源。
- 颜色对比满足浅色主题可读性，红/绿状态不只靠颜色表达。

## 测试与验证

- 构建验证：`:app:testDebugUnitTest`、`:app:assembleDebug`。
- 视觉验证：使用模拟器或真机检查首页、做饭、附近、详情、收藏、设置。
- 状态验证：
  - 数据库搜索加载与结果入场。
  - AI 生成加载、取消、失败降级。
  - 高德定位雷达、无结果、fallbackReason。
  - 收藏触觉/弹性反馈和持久化。
  - 开发者模式开关下禁用状态。
- 真机验证：定位、地图跳转、性能流畅度和触控反馈。

## 实施顺序

1. 建立新主题与基础组件，不改变业务行为。
2. 升级首页和底部导航，形成新视觉基调。
3. 升级做饭页，加入数据库/AI 两种加载动效。
4. 升级附近页，加入定位雷达和排序动效。
5. 升级详情、收藏、设置页面。
6. 清理 `MainActivity.kt` 中迁出的 Composable。
7. 跑构建和真机/模拟器视觉验证。

## 已确认决策

- 配色使用 A：明亮食欲、番茄红、阳光橙、薄荷绿。
- 动效使用 C：轻未来、共享元素感、搜索展开、结果错峰、定位雷达。
- 不采用 B 的全局黑底强霓虹风。
- 本次优先体现原生 Android 质感，而不是重做业务能力。
