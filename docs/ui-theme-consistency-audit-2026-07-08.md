# 吃点啥 Android UI 主题一致性审计

审计日期：2026-07-08

## 审计范围

- 范围：原生 Android App 的 Compose 主题、共用 UI 组件、首页、做饭推荐、附近餐厅、详情、收藏与设置页面。
- 目标：第一阶段只审计，不修改代码；识别绿色/黄色/番茄色混用导致的主题不统一问题。
- 目标主题：温和的「食谱 + 附近餐饮推荐 App」，以暖白底、柔和食欲色、少量定位绿为主。

## 证据限制

- 当前没有连接 Android 真机或模拟器，`adb devices` 未返回设备，因此本轮无法采集真实运行截图。
- 本审计基于代码、设计文档和主题令牌做证据判断；视觉结论需要第二阶段在真机或模拟器截图中复核。
- 本轮未修改 App 源码。

## 当前主题基底

项目已经有集中主题入口：

- `app/src/main/java/com/dinnerplan/chidian/ui/theme/ChiDianTheme.kt`
- `ChiDianColors` 定义了暖白画布、白色表面、暖卡片、番茄红、阳光黄、橙色、薄荷绿、AI 蓝/青、文字色。
- `ChiDianLightScheme` 把 `primary` 设为番茄红，`secondary` 设为薄荷绿。
- 文档里的设计方向也明确：番茄红用于主动作，阳光橙/暖黄用于食欲高光，薄荷绿用于附近、定位、地图数据状态。

## 主要发现

### 1. 薄荷绿已经从功能辅色变成多个页面的视觉主色

证据：

- `HomeScreen.kt` 中「今日决策」卡、按钮和餐厅入口大量使用 `MintDark`。
- `NearbyRestaurantScreen.kt` 中搜索图标、输入光标、搜索按钮、排序选中态、提示卡、距离胶囊、理由胶囊、详情按钮都使用 `MintDark` 或 `Mint.copy(...)`。
- `DetailScreens.kt` 中餐厅详情使用 `green = true` 后，Hero fallback 会切到 `MintDark -> Mint` 绿色渐变，导航按钮也是 `MintDark`。
- `SavedSettingsScreens.kt` 中餐厅收藏卡通过 `green = true` 把缩略图、标签、收藏图标切到绿色。

风险：

- 用户会感觉「附近餐厅」是另一套绿色 App，而不是同一个温和美食 App 的附近模块。
- 首页同时有番茄入口和绿色决策按钮，主动作归属不清。

建议方向：

- 保留绿色作为定位、地图、距离、营业状态等语义点缀。
- 搜索提交、主 CTA、收藏成功、确认类动作回到统一主动作色。
- 附近页面可以用暖白卡片 + 少量绿色状态，不要整页由绿色驱动。

### 2. 黄色/橙色定位不清，容易从食欲高光变成泛黄感

证据：

- `ChiDianGradients.AppetiteHero` 使用番茄红、橙色、阳光黄。
- `ChiDianGradients.NearbyHero` 使用薄荷绿、AI 青、阳光黄。
- `SurfaceWarm`、`Line`、`Sun` 都偏暖黄区间，普通卡片、分割线、标签底色叠加后容易显黄。

风险：

- 页面会在「暖和」和「发黄」之间摇摆，尤其卡片边框、标签底、Hero 高光同时出现时。
- 附近模块的绿 + 青 + 黄渐变会削弱食谱 App 的统一气质。

建议方向：

- 把暖黄限定为 Hero 氛围和少量高光，不用于大面积承载信息。
- 普通卡片保持白色或极浅暖白，边线降低饱和度。
- 附近模块避免绿青黄大渐变作为主视觉。

### 3. 仍有绕过主题的旧色值

证据：

- `MainActivity.kt` 仍定义 `private val Tomato = Color(0xFFE2533D)` 和 `private val Muted = Color(0xFF776B61)`。
- 弹窗确认按钮仍使用本地 `Tomato`，不是 `ChiDianColors.Tomato`。

风险：

- 后续统一主题时可能漏掉弹窗、Toast 周边状态或旧入口，导致“看起来已经改了，但局部还是旧色”。
- 两个番茄红值相近但不一致，会造成细微但真实的割裂。

建议方向：

- 第二阶段先清理主题绕行点，把旧色值迁回 `ChiDianColors`。
- 不改业务逻辑，只替换视觉 token 来源。

### 4. 主题 token 缺少明确角色层，页面只能靠颜色名表达语义

证据：

- 现有 token 是 `Tomato`、`Mint`、`Sun`、`Orange`、`AiBlue` 这类颜色名。
- 页面通过 `green: Boolean` 或 `tint == ChiDianColors.MintDark` 推导状态。
- `FoodChip` 直接用 `green` 参数决定番茄或薄荷。

风险：

- 颜色被页面语义绑死，后续想统一主题时，容易变成到处判断 `green`。
- “餐厅=绿色、食谱=番茄”会继续固化，和“同一个 App 的两个任务”相冲突。

建议方向：

- 第二阶段新增或整理角色 token，例如 `ActionPrimary`、`AccentLocation`、`AccentFoodWarm`、`SurfaceSubtle`、`BorderSubtle`。
- 共用组件吃角色色，不吃页面传入的颜色名。

### 5. 对比度需要在主题治理时一起处理

快速对比检查：

- 白字 / `Tomato #FF4F3E`：约 3.26:1。
- 白字 / `MintDark #1F8F5A`：约 4.09:1。
- 白字 / `AiCyan #18D9C5`：约 1.78:1。
- 白字 / `Sun #FFC857`：约 1.54:1。
- `Muted #746A61` / `SurfaceWarm #FFF6E8`：约 4.93:1。

风险：

- 番茄红和薄荷绿按钮上的白字对普通正文偏紧，尤其小字号按钮、胶囊标签或弱视场景。
- 黄色和青色不能承载白字正文，只适合作装饰、进度光带、背景高光。

建议方向：

- 第二阶段定义深一档的按钮色和浅一档的高光色。
- 黄色、青色只做装饰，不直接放白字说明。

## 页面健康概览

1. 首页：中等风险。结构清楚，但“今日决策”和附近入口偏绿，主动作色不稳定。
2. 做饭推荐页：轻中度风险。大体沿用暖色食谱风，但蔬菜标签会触发绿色芯片，AI 蓝也参与局部视觉。
3. 附近餐厅页：高风险。绿色在搜索、排序、提示、列表、操作按钮中连续出现，最像另一套主题。
4. 详情页：中高风险。食谱详情和餐厅详情通过 `green` 切换成两套 Hero/标签体系。
5. 收藏页：中等风险。收藏列表把餐厅卡切绿，分类入口仍是番茄，存在局部分裂。
6. 设置页：低到中等风险。主要沿用番茄和暖卡片，但开发者设置、表单密度需要后续视觉复核。
7. 弹窗与全局容器：中等风险。全局底色统一，但仍有旧局部色值。

## 第二阶段建议边界

优先顺序建议：

1. 先治理主题 token 和旧色值绕行点，不碰业务流。
2. 再治理共用组件：`FoodChip`、按钮、列表操作、Hero fallback、状态卡。
3. 最后按页面逐个收敛：先附近餐厅页，再首页和详情页，收藏/设置收尾。

不建议：

- 不建议一次重做所有页面。
- 不建议把餐厅模块全部去绿；绿色保留为定位/地图语义更自然。
- 不建议把黄色作为文字或按钮主背景；保留为温和食欲高光即可。
